## Architecture Overview

GoOdin is a **Hexagonal (Ports & Adapters) / Clean Architecture** enterprise Go framework/template. The entry point is `examples/main.go` or `bootstrap/main.go` (newer Uber FX-based DI variant).

### Core Lifecycle (`internal/`)

The central `App` interface (`internal/app.go`) is implemented by `BaseApp` (`internal/base_app.go`). The lifecycle flows through three hook events:

1. `OnBeforeApplicationBootstrapped` — fires before config/DB init
2. `OnAfterApplicationBootstrapped` — fires after; receives `ConfigPath` to load TOML config
3. `OnApplicationStop` — fires on shutdown

Hooks are keyed by string (`hooks.Hook[T]`); adding the same key twice returns `ErrKeyAlreadyExists`.

### Service Entry Point (`service.go`)

`service.New(BaseConfig{})` creates a `Service` wrapping `App` and a Cobra root command. Callers register lifecycle hooks, call `service.Start()`, then `service.RootCmd.ExecuteContext(ctx)`, then `service.Shutdown()`.

### Data Access

- **`pkg/datamanager`** — `DataManager` holds named `DataAdapter` implementations keyed by type (`"sql"`) and connection alias.
- **`adapters/`** — Concrete SQL adapters (Postgres raw, Postgres GORM, MySQL). Register new adapters in `adapters/register.go`.
- Access via `app.Data().Get("sql", "gorm")` which returns `*gorm.DB`, or `app.Sql("default")` for `*sql.DB`.

### External Drivers (`pkg/driver`, `drivers/`)

- **`pkg/driver.Driver`** — manages named `IDriver` instances (Init/Close lifecycle). Register via `app.Driver().RunDriver(myDriver)`. Retrieve via `app.Driver().Instance("driver-name")`.
- **`drivers/http/`** — Echo HTTP server, Swagger, request binding, API response helpers.
- **`drivers/watermill/`** — Watermill event bus (CommandBus + QueryBus + EventBus). Critical topics use SQL PubSub (durable); non-critical use go-channel (in-memory). See `drivers/watermill/register.go` for the full event topology.
- **`drivers/monitoring/`** — OpenTelemetry (traces, metrics, logs via OTLP).

### Use Cases Structure

Each feature under `use_cases/<domain>/` must follow:
```
use_cases/<domain>/
  dto/                        # Request/response structs (may be protobuf-generated)
  exception/                  # Domain-specific errors
  port/
    input/
      service/                # Input service interfaces
      message/
        listener/             # Message listener interfaces
        request/              # Outbound request interfaces (e.g. gRPC/NATS request-reply)
    output/
      repository/             # Repository interfaces
      messaging/              # Event publisher interfaces
  <domain>_service_impl.go    # Implements port/input/service interface
  <domain>_publisher_impl.go  # Implements port/output/messaging interface
  <domain>_repository_impl.go # (in repositories/ package)
```

Services use the **functional options pattern** for dependency injection:
```go
srv, err := orders.NewOrderServiceImpl(app,
    orders.WithOrderRepository(...),
    orders.WithOrderPublisher(...),
)
```

### Infrastructure (`infrastructure/`)

Connection implementations for external providers (NATS, databases). These are wired into `DataAdapter` or `IDriver` and injected at startup.

### Mocks

Generated by [Mockery](https://github.com/vekstrand/mockery) using `.mockery.yml`. All mocks go to `mocks/` — **do not edit manually**. Regenerate with `mockery` after changing interfaces.

### Configuration

TOML config loaded via Viper (`pkg/config/`). Default file: `config.toml`. Pass config path via the `AfterApplicationBootstrapped` event's `ConfigPath` field. See `config.toml` for all available sections: `[app]`, `[apis]`, `[monitoring]`, `[default_database]`, `[[databases_sql]]`.

### Observability

OpenTelemetry is set up in `drivers/monitoring/`. Tracing context is propagated per-request via `monitoring.NewTraceContext(ctx)` builder. Enable with `monitoring.enable_telemetry = true` in config.

## Key Rules

- Time values stored in DB must be `int64` (Unix timestamp).
- Interface compliance is verified with `var _ InterfaceName = (*ImplStruct)(nil)` at package level.
- Use pointer receivers consistently — do not mix value and pointer receivers on the same type.
- Avoid `init()` functions.

## Event-Driven Navigation Guide

To understand what happens when event X occurs:
1. **Find the event constant**: `grep pkg/events/` for the domain name
2. **See who reacts**: check `drivers/watermill/register.go` (the topology map + topology comment)
3. **See the full flow**: open `docs/events/flows/{flow_name}.puml`
4. **Test the chain**: run `tests/flows/{flow_name}_test.go`

## Adding a New Event — Required Checklist

Every new event **MUST** update ALL of the following. Run `make check-events` to verify 1–2.

1. `pkg/events/{domain}.go` — add the string constant with publisher/subscriber doc comment
2. `drivers/watermill/register.go` — add `eventBus.Subscribe(...)` call + update topology comment block
3. `docs/events/event_flows.puml` — update the full topology diagram
4. `use_cases/{domain}/domain/event/{event_name}.go` — the domain event struct (implements `cqrs.DomainEvent`)
5. `use_cases/{domain}/event/on_{triggering_event}_handler.go` — the event handler

## CQRS Handler Template

Every command handler that reacts to an event must follow this template:

```go
func (h *Handler) Handle(ctx context.Context, cmd Cmd) (Result, error) {
    // 1. Idempotency check — skip if already processed (Watermill may replay)
    if exists, _ := h.idem.Exists(ctx, cmd.CausationEventID); exists {
        return Result{}, nil
    }
    // 2. Domain work (inside caller's TX)
    // 3. Publish next events via SQL EventBus (writes to watermill_messages, same TX)
    // 4. Mark idempotency key (same TX)
    // TX: all steps commit together OR all roll back — no partial state
}
```

# RTK (Rust Token Killer) - Token-Optimized Commands

## Golden Rule

**Always prefix commands with `rtk`**. If RTK has a dedicated filter, it uses it. If not, it passes through unchanged. This means RTK is always safe to use.

**Important**: Even in command chains with `&&`, use `rtk`:
```bash
# ❌ Wrong
git add . && git commit -m "msg" && git push

# ✅ Correct
rtk git add . && rtk git commit -m "msg" && rtk git push
```
